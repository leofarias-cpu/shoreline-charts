<details>
  <summary><h4>When should a Button be primary, secondary, or tertiary?</h4></summary>
  <div>
    This answer depends on research to understand how frequently merchants use an action. Even so, there are patterns you can follow:
    <ul>
        <li>Primary Buttons are frequently triggered, secondary Buttons are occasionally triggered, and tertiary actions are rarely triggered.</li>
        <li>There should be only one primary action visible at a time.</li>
        <li>There should be only one secondary action in a group of actions.</li>
        <li>When an action is repeated multiple times, such as for each list item, use a tertiary Button.</li>
    </ul>
  </div>
</details>

<details>
  <summary><h4>When should a Button be neutral tertiary?</h4></summary>
  <div>
    <ul>
        <li>When the same Button is repeated, such as for each list item.</li>
        <li>When there’s a group of tertiary Buttons, they all should be neutral.</li>
        <li>When the container background uses the $bg.secondary color.</li>
    </ul>
  </div>
</details>

<details>
  <summary><h4>When should a Button be critical?</h4></summary>
  <div>
    A Button should be critical when it triggers an irreversible action. Actions should be reversible whenever possible and not require the critical color.
  </div>
  <br />
  <div>
    <strong>What to avoid:</strong>
  </div>
  <div>
    Don’t use critical colors for a Move to trash action that implies the possibility of reverting the action.
  </div>
</details>

<details>
  <summary><h4>When should a Button be normal or large?</h4></summary>
  <div>
    Use the large size for all primary Buttons. Use the normal size for secondary and tertiary Buttons that are not next to large Buttons.
  </div>
</details>

</PageSection>

<details>
  <summary><h4>Where should a Button be positioned in a container?</h4></summary>
  <div>
    Considering left-to-right interfaces, follow these recommendations:
    <ul>
        <li>Align Buttons to the right. For example, Buttons in the header of a Card should be vertically-centered and aligned to the right.</li>
        <li>When Buttons are in a horizontal list, order them from tertiary to primary. For example, a secondary Button should be to the left of a primary Button.</li>
        <li>When Buttons are in a vertical list, order them from primary to tertiary. For example, a secondary Button should be below a primary button.</li>
        <li>A secondary Button should be next to either a single primary Button or to one or more tertiary Buttons.</li>
        <li>When a tertiary Button is next to other content, ignore the Button's box to align based on the Button's icon and/or label. Use the <code>bleed</code> prop to implement this alignment.</li>
        <li>When a Button is in a form, align it to the left when it modifies content and to the right when it submits the form.</li>
    </ul>
  </div>
  <br />
  <div>
    <strong>What to avoid:</strong>
  </div>
  <div>
    <ul>
      <li>Don't position a primary Button next to a tertiary Button, with no secondary Button in between.</li>
      <li>Don't include more than one secondary Button in a list.</li>
    </ul>
  </div>
</details>


</PageSection>

<details>
  <summary><h4>When to use a disabled Button?</h4></summary>
  <div>
    <ul>
        <li>Use a disabled Button only if, in some condition, it becomes enabled. For example, a Save Button that is only enabled when edits have been made.</li>
        <li>When the reason why an action is disabled might be unclear, leave the action enabled and show a modal explaining it on trigger.</li>
    </ul>
  </div>
  <br />
  <div>
    <strong>What to avoid:</strong>
  </div>
  <div>
    <ul>
      <li>Don't display a <a href="https://admin-ui.vercel.app/components/tooltip">Tooltip</a> when hovering a disabled Button.</li>
    </ul>
  </div>
</details>

<details>
  <summary><h4>When to use a loading Button?</h4></summary>
  <div>
    <ul>
        <li>Use a loading Button after you trigger an action that usually takes more than a second and less than five seconds to process.</li>
        <li>Actions that take less than a second don't need loading or progress indicators.</li>
        <li>If the action takes more than five seconds to process, open an overlay that includes progress feedback.</li>
    </ul>
  </div>
</details>

<details>
  <summary><h4>When should a Button include a confirmation Modal?</h4></summary>
  <div>
    <ul>
        <li>When the action is irreversible. For example, a <em>Delete</em> action where it isn't possible to restore the item from the trash.</li>
        <li>When the action would take considerable time to undo. For example, a <em>Publish</em> action that takes time and can’t be canceled while in progress.</li>
    </ul>
  </div>

<details>
  <summary><h4>What should be the label of a Button?</h4></summary>
  <div>
    <ul>
        <li>Reuse existing labels.</li>
        <li>Always include a verb in the infinitive and an object. For example, <em>Create product</em>.</li>
        <li>Use a verb that reflects the exact action that the Button triggers. For example, use a <em>Publish</em> Button instead of a <em>Confirm</em> Button.</li>
        <li>Use sentence case but capitalize proper nouns.</li>
    </ul>
  </div>
  <br />
  <div>
    <strong>What to avoid:</strong>
  </div>
  <div>
    <ul>
     <li>Don’t use punctuation.</li>
     <li>Don’t use more than three words.</li>
     <li>Don’t use personal pronouns.</li>
    </ul>
  </div>
</details>

<details>
  <summary><h4>When should a Button include an icon besides a label?</h4></summary>
  <div>
    <ul>
        <li>Always use an icon to the left when the action can be represented by an icon.</li>
        <li>Always use an icon to the left when the actions are in a group.</li>
        <li>Use an external link icon to the right when the Button opens a page in a new tab.</li>
    </ul>
  </div>
</details>

<details>
  <summary><h4>When should a Button have a hidden label and only an icon visible?</h4></summary>
  <div>
    <ul>
        <li>When an icon is easily recognizable by itself. For example, the <em>Close</em>, <em>Forward</em>, and <em>Back</em> icons.</li>
        <li>When the same Button is repeated multiple times on a screen, such as for each list item.</li>
    </ul>
  </div>
  <br />
  <div>
    <strong>What to avoid:</strong>
  </div>
  <div>
    <ul>
     <li>Don’t leave a Button without an alternative label. Add it for accessibility in the implementation through the <code>aria-label</code> prop.</li>
     <li>Don’t use an icon-only Button that is not easily understood just because space is constrained.</li>
     <li>Don’t use a <a href="https://admin-ui.vercel.app/components/tooltip">Tooltip</a> to display the label.</li>
    </ul>
  </div>
</details>
